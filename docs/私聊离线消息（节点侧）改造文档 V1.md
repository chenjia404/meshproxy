# 去中心化聊天协议：私聊离线消息（节点侧，libp2p 版）V1

## 1. 目标

为现有 libp2p 聊天节点增加 **私聊离线消息能力**，通过 **libp2p stream protocol** 与 store 节点通信，实现：

* 对方离线 → 提交到 store
* 节点上线 → 拉取离线消息
* 成功处理 → 发送 ACK → store 删除

**不依赖 HTTP / HTTPS / 域名体系。**

---

## 2. 通信模型

### 2.1 角色

* **Chat Node（本节点）**
* **Store Node（libp2p peer）**

### 2.2 通信方式

全部通过 libp2p：

* `host.Connect(peer)`
* `host.NewStream(ctx, peerID, protocolID)`
* request/response 后关闭 stream

---

## 3. Protocol 定义

使用 3 个 protocol（推荐）：

```text
/chat/offline/store/1.0.0
/chat/offline/fetch/1.0.0
/chat/offline/ack/1.0.0
```

---

## 4. 编码格式（必须实现）

使用 **length-prefixed JSON**

### 帧格式：

```text
[4字节 big-endian 长度][JSON bytes]
```

### 读取规则：

1. 读4字节长度
2. 读取对应长度
3. 反序列化 JSON

禁止：

* 直接读流不分包
* 使用不定长 JSON

---

## 5. Store 节点地址获取

从用户资料中获取 **`offline_store_nodes`** 陣列。每個元素為一個 **OfflineStoreNode**：

| 欄位 | 必填 | 說明 |
|------|------|------|
| `peer_id` | 是 | Store 節點 libp2p PeerID |
| `addrs` | 否 | 可達 **multiaddr** 列表；**省略時**由本節點在網絡上 **DHT `FindPeer(peer_id)`** 解析地址後再連線（需本節點已啟用帶 DHT 的 `routing.PeerRouting`） |

示例（僅 PeerID，走網絡發現）：

```json
{
  "peer_id": "12D3KooUser",
  "offline_store_nodes": [
    {
      "peer_id": "12D3KooStoreA"
    }
  ]
}
```

示例（同時配置多地址，優先使用顯式地址，不強制 DHT）：

```json
{
  "peer_id": "12D3KooUser",
  "offline_store_nodes": [
    {
      "peer_id": "12D3KooStoreA",
      "addrs": [
        "/ip4/1.2.3.4/tcp/4001",
        "/ip4/1.2.3.4/udp/4001/quic-v1"
      ]
    }
  ]
}
```

節點側建議封裝 **`offlinestore.ConnectStorePeer(ctx, host, routing, node, ttl)`**（實作於 `internal/offlinestore/connect.go`）：

* 若 **`addrs` 非空**：`peerstore.AddAddrs` → `host.Connect`
* 若 **`addrs` 為空**：`routing.PeerRouting.FindPeer` → `peerstore.AddAddrs` → `host.Connect`

底層連線仍須滿足 **context timeout**（與第 10 節一致）。

同之前版本（保持一致），略。

## 7. 节点发送离线消息（libp2p）

## 7.1 流程

```text
sendPrivateMessage()
  → 判断对方在线
    → 在线：走原实时
    → 离线：submitOfflineMessage()
```

---

## 7.2 submitOfflineMessage 实现

### 步骤：

1. 获取收件人：

   * 公钥
   * store peer list

2. 构造明文消息

3. 加密（recipient pubkey）

4. 构造 envelope

5. 签名

6. 对每个 store：

```go
s, err := host.NewStream(ctx, storePeerID, "/chat/offline/store/1.0.0")
```

7. 写入请求帧

8. 读取响应帧

9. 关闭 stream

---

## 7.3 请求结构

```json
{
  "version": 1,
  "message": { ...envelope... }
}
```

---

## 7.4 响应结构

```json
{
  "ok": true,
  "duplicate": false,
  "store_seq": 123,
  "expire_at": 1712592000,
  "error_code": ""
}
```

---

## 7.5 错误处理

可重试：

* stream 打不开
* 超时
* EOF
* 连接失败

不可重试：

* INVALID_SIGNATURE
* TTL_TOO_LARGE
* INVALID_PAYLOAD

---

## 8. 节点拉取离线消息（libp2p）

---

## 8.1 触发时机

* 节点启动
* 网络恢复
* 定时任务

---

## 8.2 拉取流程

```text
for each store:
    lastSeq = loadLocalAckSeq(store)

    loop:
        fetch(after_seq=lastSeq)

        逐条处理

        更新 ack_seq

        发送 ACK

        更新本地游标
```

---

## 8.3 fetch 请求

protocol:

```text
/chat/offline/fetch/1.0.0
```

request:

```json
{
  "version": 1,
  "recipient_id": "peerB",
  "after_seq": 120,
  "limit": 100
}
```

---

## 8.4 fetch 响应

```json
{
  "ok": true,
  "items": [...],
  "has_more": true,
  "error_code": ""
}
```

---

## 8.5 单条处理流程（必须严格）

顺序：

1. recipient 校验
2. 验签
3. 去重检查
4. 解密
5. 入库
6. 标记 processed

任何一步失败：

→ 停止推进 ACK

---

## 9. ACK（libp2p）

---

## 9.1 protocol

```text
/chat/offline/ack/1.0.0
```

---

## 9.2 请求

```json
{
  "version": 1,
  "recipient_id": "peerB",
  "device_id": "device-1",
  "ack_seq": 123,
  "acked_at": 1710001000,
  "signature": { ... }
}
```

---

## 9.3 响应

```json
{
  "ok": true,
  "deleted_until_seq": 123
}
```

---

## 9.4 ACK 规则

* 只能 ACK 连续成功的最大序号
* ACK 成功后更新本地游标
* ACK 失败必须重试

---

## 10. libp2p 连接策略（重要）

---

## 10.1 不要自己维护长连接

每次：

```go
NewStream → 请求 → 响应 → Close
```

底层连接 libp2p 会复用。

---

## 10.2 超时控制

必须加 context timeout：

```go
ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
```

---

## 10.3 并发策略

建议：

* fetch：串行（保证顺序）
* submit：可并发多个 store
* ack：串行 per store

---

## 11. 节点本地状态（不变）

* processed_msg_id
* last_acked_seq (per store)
* pending_submit_tasks
* pending_ack_tasks

---

## 12. 重试机制（libp2p 特化）

---

## 12.1 提交重试

触发：

* stream error
* timeout
* connection failed

策略：

指数退避

---

## 12.2 ACK 重试

关键：

* ACK 可以重复发送
* 必须幂等
* 永远保留最大 ack_seq

---

## 13. 安全要求（新增 libp2p 相关）

### 13.1 必须校验 peer_id

连接 store 时：

```go
host.Connect(peerInfo)
```

libp2p 会校验 peer identity。

禁止：

* 连接后忽略 peer id
* 信任未认证 peer

---

### 13.2 每条消息仍需签名验证

libp2p 安全通道 ≠ 消息可信
必须继续验签。

---

## 14. 错误码（统一）

INVALID_SIGNATURE
TTL_TOO_LARGE
RECIPIENT_NOT_FOUND
DUPLICATE

## 15. 必须新增模块

---

### 15.1 Libp2pStoreClient

职责：

* connect peer
* open stream
* send request
* read response

接口：

```text
StoreMessage(peerID, payload)
FetchMessages(peerID, afterSeq)
AckMessages(peerID, ackSeq)
```

---

### 15.2 StreamCodec

职责：

* writeFrame()
* readFrame()

---

### 15.3 OfflineMessageService

整合：

* submit
* fetch
* ack
* retry

---

## 16. 关键实现约束

必须满足：

1. 所有 libp2p stream 必须 close
2. 所有请求必须有 timeout
3. 所有 response 必须校验 ok 字段
4. 所有消息必须验签
5. 所有 msg_id 必须幂等
6. ACK 必须严格递增
7. store_seq 必须顺序处理
8. 本地状态必须持久化

---

## 17. 开发顺序（强制）

1. StreamCodec
2. libp2p store client
3. 加密+签名模块
4. submit
5. fetch
6. ack
7. retry
8. 状态持久化
9. 测试

---

## 18. 测试重点（libp2p 特有）

必须覆盖：

* peer connect 失败
* stream open 失败
* stream 半关闭
* response 不完整
* 重复消息
* ACK 丢失
* 网络抖动恢复
* 多次 fetch 不重复入库

---

## 19. 非目标（再次强调）

不要实现：

* HTTP 接口
* store 服务端逻辑
* 长连接流同步
* gossip/广播离线消息
* 群离线

---

# 最终一句话总结（给 Codex 理解）

> 实现一个基于 libp2p stream 的离线消息 RPC 客户端，支持 store/fetch/ack 三个协议，使用 length-prefixed JSON 编码，保证消息加密、签名、幂等、顺序处理和 ACK 删除机制。
