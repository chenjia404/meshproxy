# 去中心化公开频道协议与本地存储设计（最终版 v1，结合 mesh-proxy 实际情况修订）

## 1. 文档定位

这份文档描述的是 **mesh-proxy 后续可实现的“去中心化公开频道”子系统**，不是当前已经存在的 `meshserver` 中心化频道接口本身。

这次修订的目标不是推翻原始设计，而是把它和仓库里的现实约束对齐，避免后续实现时出现“两套完全不同的频道语义”。

对齐后的总体原则如下：

* 保留“公开频道、单 owner 写入、消息有永久编号、当前状态覆盖”的核心目标
* 明确它与当前 `meshserver` 频道是两个子系统，不共用协议命名空间
* 尽量复用项目现有的同步习惯：`last_seq`、`last_seen_seq`、`after_seq`
* 时间字段统一使用 **Unix 秒级时间戳**
* 本地存储继续使用 SQLite 友好的整数主键做关联优化
* 传输层尽量贴近现有 `meshserver` 的单一 session stream + 请求响应 + 异步事件模型

---

## 2. 项目现状约束

结合当前仓库，必须先承认下面这些事实：

### 2.1 当前已落地的频道能力是 `meshserver`

当前项目里真正已经跑通的“频道”能力在：

* `internal/meshserver/sessionv1/session.pb.go`
* `internal/meshserver/client.go`
* `internal/api/local_api.go`
* `sdk/meshserver.go`

这套能力的关键特征是：

* 协议为 `/meshserver/session/1.0.0`
* 频道 ID / Space ID 在协议层是 **uint32**
* 订阅接口使用 `last_seen_seq`
* 增量同步接口使用 `after_seq`
* 服务端推送使用异步 `MessageEvent`

因此，新设计如果完全绕开 `seq` 语义，后续实现会和现有前端、SDK、同步逻辑严重割裂。

### 2.2 当前项目已有 libp2p host / discovery / gossip 基础设施

仓库里已经有：

* libp2p host
* DHT / bootstrap / discovery
* `internal/p2p/gossip.go` 中的 GossipSub 封装

但当前业务聊天并不是依赖“每条业务消息都通过 gossip 全网扩散”来完成同步。

因此，对去中心化公开频道的要求应当是：

* **PubSub 可以作为加速层**
* **不能作为唯一正确性来源**

换句话说，即使某个节点没收到 PubSub 推送，也必须能依靠拉取把数据补齐。

### 2.3 当前项目的消息类型已经不是只有纯文本

现有 `meshserver` 消息模型已经支持：

* `TEXT`
* `IMAGE`
* `FILE`
* `SYSTEM`

而且媒体对象已经有这些常见字段：

* `media_id`
* `blob_id`
* `sha256`
* `url`
* `inline_data`
* `mime_type`
* `size`

所以去中心化公开频道如果只设计成 `text + files[]`，实现层最终还是得再补一层图片语义。  
更贴近项目实际的做法是：**协议层直接保留 `text / image / file / deleted / system` 这类消息类型习惯**。

### 2.4 时间统一用秒

本次修订明确约定：

* `created_at`
* `updated_at`
* `published_at`
* `synced_at`

统一使用：

```text
Unix timestamp in seconds
```

不使用毫秒。  
如果未来某个适配层接到毫秒时间戳，应在适配边界转换为秒后再落库和传递。

## 3. 与当前 `meshserver` 频道的关系

为了避免实现时混淆，v1 先明确边界。

### 3.1 两个系统不是一回事

`meshserver` 频道：

* 是中心化服务端模型
* 由服务端分配 `channel_id uint32`
* 权限来自服务端 ACL / 成员关系
* 当前消息模型更接近“服务端追加事件流”

本文档中的去中心化公开频道：

* 是点对点可复制内容模型
* 由频道 owner 生成对外 `channel_id`
* 权威来自 owner 签名
* 消息对象采用“当前状态覆盖 + 墓碑删除”

### 3.2 API 命名空间必须分开

不要把去中心化公开频道塞进：

```text
/api/v1/meshserver/...
```

建议单独使用：

```text
/api/v1/public-channels/...
```

这样前端、SDK、协议调试和权限语义都会更清晰。

### 3.3 适合复用的，是同步习惯而不是 ID 体系

应复用的部分：

* `last_seq`
* `last_seen_seq`
* `after_seq`
* `next_after_seq`
* 单一 session stream + request/response + async event

不应硬复用的部分：

* `meshserver` 的 `uint32 channel_id`
* `space_id`
* 服务端成员权限体系

## 4. v1 核心结论

## 4.1 频道标识

协议层对外唯一标识仍然使用：

```text
channel_id = UUIDv7
```

例如：

```text
0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123
```

保留 UUIDv7 的原因：

* 不依赖中心分配器
* 时间有序，便于索引和排查
* 与 `meshserver` 的数值型 `channel_id` 明确区分

## 4.2 同步游标统一收敛到 `seq`

原设计里的 `channel_revision` 在本项目里建议直接收敛成：

```text
seq
```

也就是：

* 频道级全局单调递增游标统一命名为 `last_seq`
* 客户端订阅使用 `last_seen_seq`
* 客户端增量拉取使用 `after_seq`
* 服务端返回 `next_after_seq`

这样做的原因是：

* 与现有 `meshserver` API 一致
* 前端和 SDK 更容易复用
* 本地同步状态字段更统一

## 4.3 版本与同步分离

同一条消息的“谁更新更新”仍然靠：

```text
version
```

频道增量同步靠：

```text
seq
```

也就是说：

* `version` 解决同一消息对象的新旧判断
* `seq` 解决频道范围内的增量同步与补洞

两者不要混用。

## 4.4 单 owner、单写者假设

v1 仍然只支持：

* 一个频道一个 owner
* 只有 owner 可以写
* 不做多管理员
* 不做 owner 转移

同时再补一条更贴近工程现实的约束：

* **v1 按“单活写设备”假设实现**

也就是 owner 虽然可以多设备持有同一身份，但同一时刻最好只有一个设备执行公开频道写入。  
否则在完全去中心化情况下，会出现：

* `message_id` 抢号
* `last_seq` 并发冲突
* 离线后重放顺序不稳定

这类问题需要额外的租约或写令牌协议，v1 不做。

## 4.5 PubSub 只做提示，不做最终正确性来源

v1 的正确性必须满足：

* 没有 PubSub 推送也能同步成功
* 漏掉实时推送后，可通过 `after_seq` 补齐

因此：

* PubSub 负责“快速知道有更新了”
* 真正补齐数据仍靠 session stream / pull sync

---

## 5. v1 范围

### 5.1 v1 要支持

* owner 创建频道
* owner 更新频道资料
* owner 发布消息
* owner 修改消息
* owner 删除消息
* 订阅节点只读同步
* 首次只拉最新一页
* 向上翻页拉历史
* 用 `seq` 做增量同步和断点续传

### 5.2 v1 明确不做

* owner 转移
* 多写者
* 消息完整编辑历史
* 全量 Merkle root 同步
* 反应、引用、线程、置顶、已读回执
* 文件物理擦除承诺
* 所有副本都保存完整历史

## 6. 数据模型

## 6.1 频道资料对象 `ChannelProfile`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "owner_peer_id": "12D3KooWxxxx",
  "owner_version": 1,
  "name": "技术频道",
  "avatar": {
    "cid": "bafy...",
    "media_id": "",
    "mime": "image/png",
    "size": 24567,
    "width": 512,
    "height": 512
  },
  "bio": "技术分享",
  "profile_version": 1,
  "created_at": 1710000000,
  "updated_at": 1710000000,
  "signature": "base64sig"
}
```

字段说明：

* `channel_id`：协议层唯一 ID，UUIDv7
* `owner_peer_id`：owner 的 libp2p peer id
* `owner_version`：owner 版本号，v1 固定为 `1`
* `profile_version`：频道资料版本号
* `created_at` / `updated_at`：秒级时间戳
* `signature`：owner 对资料对象的签名

## 6.2 频道头对象 `ChannelHead`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "owner_peer_id": "12D3KooWxxxx",
  "owner_version": 1,
  "last_message_id": 200,
  "profile_version": 3,
  "last_seq": 168,
  "updated_at": 1710009999,
  "signature": "base64sig"
}
```

字段说明：

* `last_message_id`：当前频道已分配的最大消息号
* `profile_version`：当前资料版本
* `last_seq`：频道当前最大的同步游标
* `updated_at`：Head 最后更新时间，秒级

## 6.3 媒体对象

为了兼容现有项目的消息/文件处理习惯，定义两个媒体对象。

### `ChannelImage`

```json
{
  "cid": "bafy...",
  "media_id": "",
  "blob_id": "",
  "sha256": "abc...",
  "url": "",
  "mime": "image/jpeg",
  "size": 123456,
  "width": 1080,
  "height": 720,
  "name": "image.jpg"
}
```

### `ChannelFile`

```json
{
  "cid": "bafy...",
  "media_id": "",
  "blob_id": "",
  "sha256": "abc...",
  "url": "",
  "mime": "application/pdf",
  "size": 2481234,
  "name": "report.pdf"
}
```

说明：

* 当前实现里通过 `blob_id` 承载 IPFS CID
* 如果业务侧需要记录 CID，请直接持久化 `blob_id`
* 如果后续需要桥接现有 `meshserver` / 本地媒体缓存能力，可临时带上 `media_id`
* 默认推荐通过当前节点本地 `/ipfs/{blob_id}/...` 路径访问资源
* `url` 只作为本地网关或缓存层的展示字段，不是权威标识

## 6.4 消息对象 `ChannelMessage`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "message_id": 15,
  "version": 2,
  "seq": 121,
  "owner_version": 1,
  "creator_peer_id": "12D3KooWxxxx",
  "author_peer_id": "12D3KooWxxxx",
  "created_at": 1710000000,
  "updated_at": 1710000300,
  "is_deleted": false,
  "message_type": "file",
  "content": {
    "text": "这是更新后的内容",
    "images": [],
    "files": [
      {
        "cid": "bafy...",
        "media_id": "",
        "blob_id": "",
        "sha256": "abc...",
        "url": "",
        "mime": "application/pdf",
        "size": 2481234,
        "name": "report.pdf"
      }
    ]
  },
  "signature": "base64sig"
}
```

字段说明：

* `message_id`：频道内永久编号，从 `1` 开始
* `version`：同一消息的当前版本号
* `seq`：该版本进入频道变化序列时占用的全局游标
* `created_at`：首次创建时间，永不改变
* `updated_at`：当前版本写入时间，秒级
* `message_type`：`text` / `image` / `file` / `system` / `deleted`

## 7. 本地存储设计

考虑到项目里已有多个 SQLite store，v1 本地存储建议继续走 SQLite，并用整数主键优化关联。

## 7.1 数据库文件拆分建议

去中心化公开频道建议使用单独数据库文件：

```text
{data_dir}/public_channels.db
```

而不是继续塞进现有 `chat.db`。

原因：

* 公开频道的消息量、历史长度和同步索引规模，预期会明显大于普通私聊状态
* 公开频道会持续产生 `messages / changes / sync_state`，更容易把主聊天库做大
* 单独建库后，备份、清理、迁移、修复和 `VACUUM` 都更容易控制
* 公开频道和私聊/群聊之间通常不需要跨库强事务，一体化收益不高

结合当前项目现状，更推荐的落地方式是：

* `chat.db`：继续承载现有私聊、群聊、本地聊天相关数据
* `public_channels.db`：承载去中心化公开频道数据
* 媒体文件：继续走文件系统缓存、IPFS 或对象引用，不把大二进制直接塞进 SQLite

### 7.1.1 为什么不是“每个频道一个 sqlite”

v1 不建议做成：

```text
channel-<uuid>.db
```

这类每频道单独分库方案。

原因：

* 文件数会快速增长，备份和迁移复杂度明显上升
* 打开连接、做 schema 迁移、做索引检查都更麻烦
* 跨频道聚合查询会很差，比如“我本地有哪些公开频道”
* 当前阶段还没有证据表明按频道分库能带来足够收益

因此 v1 推荐：

* **一个公开频道子系统一个库**
* **库内多张表按 `channel_db_id` 隔离**

## 7.2 `public_channels`

```sql
CREATE TABLE public_channels (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_id TEXT NOT NULL UNIQUE,
  owner_peer_id TEXT NOT NULL,
  owner_version INTEGER NOT NULL,
  name TEXT NOT NULL,
  avatar_json TEXT NOT NULL DEFAULT '',
  bio TEXT NOT NULL DEFAULT '',
  profile_version INTEGER NOT NULL,
  last_message_id INTEGER NOT NULL DEFAULT 0,
  last_seq INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  profile_signature TEXT NOT NULL,
  head_signature TEXT NOT NULL DEFAULT ''
);
```

说明：

* `id` 是本地整数主键
* `channel_id` 才是协议层 ID
* `avatar_json` 直接存结构化 JSON，便于 v1 快速落地

## 7.3 `public_channel_messages`

```sql
CREATE TABLE public_channel_messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_db_id INTEGER NOT NULL,
  message_id INTEGER NOT NULL,
  version INTEGER NOT NULL,
  seq INTEGER NOT NULL,
  owner_version INTEGER NOT NULL,
  creator_peer_id TEXT NOT NULL,
  author_peer_id TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  is_deleted INTEGER NOT NULL DEFAULT 0,
  message_type TEXT NOT NULL,
  content_json TEXT NOT NULL DEFAULT '',
  signature TEXT NOT NULL,
  UNIQUE(channel_db_id, message_id),
  UNIQUE(channel_db_id, seq)
);
```

说明：

* `content_json` 里保存 `text/images/files`
* v1 优先保证实现简单和同步正确，先不拆成过多子表
* 如果后续媒体查询压力上来，再单独拆媒体表

## 7.4 `public_channel_changes`

```sql
CREATE TABLE public_channel_changes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_db_id INTEGER NOT NULL,
  seq INTEGER NOT NULL,
  change_type TEXT NOT NULL,
  message_id INTEGER,
  version INTEGER,
  is_deleted INTEGER,
  profile_version INTEGER,
  created_at INTEGER NOT NULL,
  UNIQUE(channel_db_id, seq)
);
```

## 7.5 `public_channel_sync_state`

```sql
CREATE TABLE public_channel_sync_state (
  channel_db_id INTEGER PRIMARY KEY,
  last_seen_seq INTEGER NOT NULL DEFAULT 0,
  last_synced_seq INTEGER NOT NULL DEFAULT 0,
  latest_loaded_message_id INTEGER NOT NULL DEFAULT 0,
  oldest_loaded_message_id INTEGER NOT NULL DEFAULT 0,
  subscribed INTEGER NOT NULL DEFAULT 0,
  updated_at INTEGER NOT NULL DEFAULT 0
);
```

字段说明：

* `last_seen_seq`：本地已经确认看到的最大 `seq`
* `last_synced_seq`：本地已经完整落库的最大 `seq`
* `latest_loaded_message_id`：当前内存/页面已加载的最大消息号
* `oldest_loaded_message_id`：当前已加载的最小消息号

## 7.6 推荐索引

```sql
CREATE INDEX idx_public_channels_owner ON public_channels(owner_peer_id);
CREATE INDEX idx_public_channel_messages_channel_msg ON public_channel_messages(channel_db_id, message_id DESC);
CREATE INDEX idx_public_channel_messages_channel_seq ON public_channel_messages(channel_db_id, seq ASC);
CREATE INDEX idx_public_channel_changes_channel_seq ON public_channel_changes(channel_db_id, seq ASC);
```

### 7.6.1 可选运行参数

如果后续确认公开频道写入频繁，可以单独对 `public_channels.db` 评估以下 SQLite 参数：

* `PRAGMA journal_mode=WAL`
* `PRAGMA busy_timeout = 15000`
* 合理的自动 checkpoint 策略

这样即使公开频道同步比较活跃，也不会把其它聊天库的写压力和 checkpoint 行为绑在一起。

## 8. 核心规则

## 8.1 一个 owner 可以创建多个频道

同一个 `owner_peer_id` 可以创建多个 `channel_id`，彼此完全独立。

## 8.2 消息编号规则

`message_id` 只在单频道内自增：

* 第一条消息 `message_id = 1`
* 第二条消息 `message_id = 2`

消息逻辑主键是：

```text
(channel_id, message_id)
```

## 8.3 `version` 规则

创建消息：

* `version = 1`

修改消息：

* `version = old_version + 1`

删除消息：

* `version = old_version + 1`

## 8.4 `seq` 规则

以下任何操作都会消耗一个新的 `seq`：

* 创建消息
* 修改消息
* 删除消息
* 修改频道资料

例如：

* 初始 `last_seq = 0`
* 发布第一条消息后 `last_seq = 1`
* 编辑消息 1 后 `last_seq = 2`
* 更新频道简介后 `last_seq = 3`

## 8.5 排序规则

频道阅读视图固定按：

```text
ORDER BY message_id DESC
```

也就是说：

* 新消息排前面
* 编辑旧消息不会顶到最前面
* 删除旧消息也不会改变位置

## 8.6 新旧判断规则

同一消息对象的新旧判断优先级：

1. 比较 `version`
2. `version` 相同则比较签名对象是否一致
3. `updated_at` 只用于展示和排障，不作为权威冲突解决字段

频道同步进度判断只看：

```text
seq
```

## 9. 写入规则

## 9.1 创建频道

创建频道时：

* 生成 `channel_id = UUIDv7`
* 设置 `owner_peer_id`
* 设置 `owner_version = 1`
* 设置 `profile_version = 1`
* 设置 `last_message_id = 0`
* 设置 `last_seq = 0`
* `created_at = now_sec`
* `updated_at = now_sec`
* 本地 `public_channel_sync_state.subscribed = 1`
* 生成 `ChannelProfile` 签名
* 生成 `ChannelHead` 签名

## 9.2 修改频道资料

修改资料时：

* `profile_version = old_profile_version + 1`
* `last_seq = old_last_seq + 1`
* `updated_at = now_sec`
* 追加一条 `ChannelChange(change_type = "profile")`
* 重签 `ChannelProfile`
* 重签 `ChannelHead`

## 9.3 创建消息

创建时：

* `message_id = head.last_message_id + 1`
* `version = 1`
* `seq = head.last_seq + 1`
* `created_at = now_sec`
* `updated_at = now_sec`
* `author_peer_id = owner_peer_id`
* `creator_peer_id = owner_peer_id`
* 更新 `head.last_message_id`
* 更新 `head.last_seq`
* 写入 `ChannelChange(change_type = "message")`

## 9.4 修改消息

修改时：

* `message_id` 不变
* `version = old_version + 1`
* `seq = head.last_seq + 1`
* `created_at` 不变
* `updated_at = now_sec`
* `author_peer_id = owner_peer_id`
* 重写当前内容
* 更新 `head.last_seq`
* 写入 `ChannelChange(change_type = "message")`

## 9.5 删除消息

删除时：

* `message_id` 不变
* `version = old_version + 1`
* `seq = head.last_seq + 1`
* `created_at` 不变
* `updated_at = now_sec`
* `is_deleted = true`
* `message_type = "deleted"`
* `content.text = ""`
* `content.images = []`
* `content.files = []`
* 更新 `head.last_seq`
* 写入 `ChannelChange(change_type = "message")`

## 10. owner 设备不持有全量数据时的规则

这是原设计里最容易在实现时踩坑的地方，这里补充成更可执行的版本。

## 10.1 发布新消息前的最小前置条件

owner 设备至少要先拿到：

* 最新 `ChannelHead`
* 其中的 `last_message_id`
* 其中的 `last_seq`

否则无法安全分配新的 `message_id` 和 `seq`。

## 10.2 修改或删除旧消息前的最小前置条件

owner 设备必须先拉取该消息当前版本，确认：

* 当前 `version`
* 当前 `is_deleted`
* 当前 `message_type`

然后才能继续写入。

## 10.3 并发写入限制

如果 owner 同时在两台设备上发消息，且两边看到的 `last_message_id` 相同，就可能产生抢号。

因此 v1 的工程要求是：

* 默认按单活写设备实现
* 若检测到 head 已被其它设备推进，应拒绝本次写入并要求先同步

可以理解为：

```text
先 sync head，再写
```

## 11. 签名与验签规则

## 11.1 签名方式

建议直接复用项目已有的 libp2p peer 身份签名习惯：

* 私钥签名
* peer id 提取公钥验签
* 签名字节做 Base64 存储

这与现有：

* `internal/binding/sign_libp2p.go`

的思路一致。

## 11.2 canonical 序列化规则

不要直接对无序 map 做 JSON 签名。  
建议像现有 `binding` 一样，使用固定字段顺序生成 canonical bytes，再签名。

## 11.3 资料签名覆盖字段

```text
channel_id
owner_peer_id
owner_version
name
avatar
bio
profile_version
created_at
updated_at
```

## 11.4 Head 签名覆盖字段

```text
channel_id
owner_peer_id
owner_version
last_message_id
profile_version
last_seq
updated_at
```

## 11.5 消息签名覆盖字段

```text
channel_id
message_id
version
seq
owner_version
creator_peer_id
author_peer_id
created_at
updated_at
is_deleted
message_type
content
```

## 11.6 验签规则

收到资料后：

* `channel_id` 必须是合法 UUIDv7
* `signature` 必须由 `owner_peer_id` 验签通过

收到 Head 后：

* `last_seq >= last_message_id` 不做强约束，但 `last_seq` 不能倒退
* `signature` 必须由 `owner_peer_id` 验签通过

收到消息后：

* `author_peer_id` 在 v1 必须等于当前 `owner_peer_id`
* `owner_version` 在 v1 固定为 `1`
* `signature` 必须由 `owner_peer_id` 验签通过

## 12. 同步设计

v1 采用：

> 当前状态同步 + `seq` 增量补齐 + 历史按需翻页

## 12.1 首次进入频道

首次进入频道时只拉：

1. `ChannelProfile`
2. `ChannelHead`
3. 最新一页消息

不拉全量历史。

## 12.2 首次进入的推荐流程

### 第 1 步

拉取 `ChannelProfile`

### 第 2 步

拉取 `ChannelHead`，记住：

```text
R = head.last_seq
```

### 第 3 步

拉最新一页消息，例如最近 20 条。

### 第 4 步

建立订阅，并携带：

```text
last_seen_seq = R
```

### 第 5 步

如果订阅建立前后存在竞态，再执行：

```text
sync_channel(after_seq = R)
```

把 `R` 之后的变更补齐。

## 12.3 历史翻页

仍然使用：

```text
before_message_id
```

例如当前拿到了 `200..181`，继续向上拉：

* `180..161`

返回仍按 `message_id DESC`。

## 12.4 增量同步

增量同步统一使用：

* `after_seq`
* `limit`

返回：

* `items`
* `next_after_seq`
* `has_more`
* `current_last_seq`

返回示例：

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "current_last_seq": 168,
  "has_more": true,
  "next_after_seq": 122,
  "items": [
    {
      "seq": 121,
      "change_type": "message",
      "message_id": 201,
      "version": 1,
      "is_deleted": false,
      "created_at": 1710010000
    }
  ]
}
```

## 12.5 客户端本地状态

客户端建议维护：

* `last_seen_seq`
* `last_synced_seq`
* `latest_loaded_message_id`
* `oldest_loaded_message_id`

用途：

* 订阅续传
* 断点增量同步
* 历史翻页

## 13. 传输层设计

## 13.1 推荐传输模型

结合当前仓库的实际代码风格，v1 更建议采用：

> 单一 long-lived session stream + request/response + async event

而不是“每个动作一个完全独立协议 + 每次临时开关 stream”的风格。

原因：

* 与 `internal/meshserver/client.go` 的 pending request 模型更接近
* 更利于复用现有客户端抽象
* 断线重连、事件推送、订阅状态维护更自然

## 13.2 PubSub topic

```text
/meshchat/public-channel/{channel_id}/v1
```

例如：

```text
/meshchat/public-channel/0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123/v1
```

PubSub 规则：

* PubSub 只推送轻量 change 通知
* PubSub 不做完整历史同步
* 客户端漏掉通知后，必须仍可通过 `sync_channel(after_seq=...)` 补齐

## 13.3 统一 stream protocol

所有点对点请求响应统一走一个 stream protocol：

```text
/meshchat/public-channel/rpc/1.0.0
```

## 13.4 统一 RPC 格式

统一 RPC 请求格式：

```json
{
  "request_id": "uuid",
  "method": "...",
  "body": {}
}
```

字段说明：

* `request_id`：请求唯一 ID，用于请求响应对应
* `method`：调用的方法名
* `body`：方法参数对象

响应格式建议为：

```json
{
  "request_id": "uuid",
  "ok": true,
  "error": "",
  "body": {}
}
```

## 13.5 建议请求动作

至少包括：

* `get_channel_profile`
* `get_channel_head`
* `get_channel_messages`
* `get_channel_message`
* `sync_channel`
* `subscribe_channel`
* `unsubscribe_channel`
* `create_channel`
* `update_channel_profile`
* `create_channel_message`
* `update_channel_message`
* `delete_channel_message`

## 13.6 PubSub change 通知

推荐 v1 的 PubSub 消息只发送轻量 change：

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "seq": 121,
  "change_type": "message",
  "message_id": 201,
  "version": 1,
  "is_deleted": false,
  "profile_version": null,
  "created_at": 1710010000
}
```

资料变更示例：

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "seq": 122,
  "change_type": "profile",
  "message_id": null,
  "version": null,
  "is_deleted": null,
  "profile_version": 4,
  "created_at": 1710010100
}
```

客户端收到后应：

* 先比较本地 `last_seen_seq`
* 必要时调用 `sync_channel(after_seq=...)`
* 不直接把 PubSub 当作完整正文来源

## 13.7 异步事件

订阅建立后，远端可以主动推送：

* `channel_change`
* `channel_head_changed`

事件本身只需携带轻量信息，客户端再按需拉正文。

## 13.8 PubSub 的角色

若要使用 PubSub，建议只做：

* 频道更新提示
* 数据源发现提示

不做：

* 完整历史同步
* 唯一同步来源

## 14. 节点发现与副本模型

## 14.1 角色定义

### owner 节点

* 唯一权威写入者
* 负责签名
* 提供最新状态

### 订阅副本节点

* 只读保存部分或全部频道数据
* 可向其它节点提供自己已有的数据

### 临时读取节点

* 未长期订阅
* 只按需拉一部分数据

## 14.2 副本节点不要求有完整历史

允许节点只缓存：

* 最新一页
* 最近 N 页
* 某个历史窗口
* 完整频道

协议必须允许返回：

* 部分数据
* 无此数据
* 建议改向其它节点继续拉取

## 14.3 数据源发现建议

推荐优先级：

1. owner 节点
2. 已知订阅副本
3. DHT / provider advertisement 找到的候选节点

## 16. 与当前项目实现最相关的落地建议

## 16.1 建议新增独立模块

建议新增：

```text
internal/publicchannel
```

至少拆成：

* `store.go`
* `service.go`
* `protocol.go`
* `sync.go`
* `types.go`

## 16.2 建议复用现有能力

可以直接复用的部分：

* libp2p host
* DHT / discovery
* GossipSub 实例
* peer id 签名验签方式
* 本地 HTTP API 风格
* SDK 的 service wrapper 风格

## 16.3 不建议直接复用的部分

不建议直接拿来套的部分：

* `meshserver/sessionv1` 的 protobuf 频道定义
* `meshserver` 的 `uint32 channel_id`
* `space / member / role` 权限模型

原因很简单：  
它们描述的是中心化服务端模型，不是 owner 签名的去中心化公开频道模型。

## 16.4 前端聚合展示建议

如果控制台以后要同时展示：

* `meshserver` 频道
* 去中心化公开频道

建议前端只在展示层做统一 view model，例如：

* `kind`
* `id`
* `name`
* `last_seq`
* `can_send`
* `source`

不要强行把两套底层协议揉成一个底层 ID 体系。

---

## 17. 最终结论

结合当前项目，v1 最重要的收敛点有四个：

1. 去中心化公开频道必须和 `meshserver` 中心化频道分成两个子系统
2. 同步游标统一使用 `seq / last_seen_seq / after_seq`，不要再额外引入一套 `channel_revision` 术语
3. 时间统一使用秒级时间戳
4. 传输层优先复用当前项目已经成熟的 session stream + 请求响应 + 异步事件模式，PubSub 只做加速层

这四点定下来后，后续无论是本地存储、SDK、控制台前端还是 libp2p 协议实现，都会顺很多。
