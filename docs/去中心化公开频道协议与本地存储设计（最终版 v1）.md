
# 去中心化公开频道协议与同步设计（最终版 v1）

## 1. 目标

在去中心化聊天系统中实现“公开频道”能力，要求如下：

* 一个用户（一个 `peer_id`）可以创建多个频道
* 每个频道的对外唯一标识符使用 **UUIDv7**
* 每个频道只有一个 owner（管理员）
* 只有 owner 可以：

  * 发布消息
  * 修改消息
  * 删除消息
  * 修改频道资料
* 消息是公开的
* 消息支持：

  * 纯文本
  * 文件 + 文本
* 每条消息在单个频道内有永久 `message_id`
* `message_id` 从 `1` 开始自增
* 频道查看时按 `message_id DESC` 排序
* 不保留完整事件日志
* 消息采用“当前状态覆盖”模型
* 底层使用 **libp2p** 作为节点之间的通信、同步和订阅网络
* 订阅节点可以收到最新内容推送，也可以从其它订阅节点拉数据
* 数据库层为了减小空间占用，`channels` 使用内部整数主键，`channel_messages` 使用该整数主键关联
* **预留 owner 转移接口，但 v1 不实现 owner 转移**

---

# 2. v1 核心结论

## 2.1 频道标识

### 协议层 / 网络层

频道唯一标识：

```text
channel_id = UUIDv7
```

例如：

```text
0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123
```

### 数据库层

本地数据库额外使用内部整数主键：

```text
channels.id BIGINT UNSIGNED AUTO_INCREMENT
```

这个整数主键只用于本地数据库关联优化，**不能用于网络协议**。

---

## 2.2 owner 模型

* 一个频道在任意时刻只有一个 owner
* owner 用 `owner_peer_id` 表示
* v1 中 owner 不可转移
* v1 不做多管理员
* v1 不做管理员权限分级

但是为了以后升级：

* 预留 `owner_version`
* 预留 owner 转移接口
* 预留 owner 转移数据结构
* v1 中固定 `owner_version = 1`

---

## 2.3 消息模型

每条消息采用 **状态覆盖模型**：

* 创建消息时生成一条消息记录
* 修改消息时，直接覆盖该消息当前内容并提升版本号
* 删除消息时，不删除记录，而是覆盖为墓碑状态
* 不保留完整编辑历史
* 不保留完整删除历史
* 不保留完整事件流

---

## 2.4 新旧判断

v1 消息新旧只依赖：

```text
version
```

不依赖 `updated_at`。

原因：

* 频道查看按 `message_id DESC`
* 编辑旧消息不应改变其列表位置
* `version` 比时间戳更适合作为协议层权威版本字段

---

## 2.5 同步核心

v1 不使用全量 `messages_root` 作为核心同步依据，因为 owner 某个设备可能没有频道全部数据，无法可靠维护全量根哈希。

v1 改用：

* `latest_message_id`
* `channel_revision`
* `channel_changes`

其中：

* `channel_revision` 表示频道变更序号
* `channel_changes` 是轻量变更索引，不是完整事件日志

---

# 3. 数据模型

## 3.1 频道资料对象 `ChannelProfile`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "owner_peer_id": "12D3KooWxxxx",
  "owner_version": 1,
  "name": "技术频道",
  "avatar": {
    "cid": "bafy...",
    "mime": "image/png",
    "size": 24567,
    "width": 512,
    "height": 512
  },
  "bio": "技术分享",
  "profile_version": 1,
  "created_at": 1710000000,
  "updated_at": 1710000000,
  "signature": "base64sig"
}
```

### 字段说明

* `channel_id`：频道对外唯一 ID，UUIDv7
* `owner_peer_id`：频道当前 owner
* `owner_version`：owner 版本号，v1 固定为 `1`
* `name`：频道名
* `avatar`：头像文件引用
* `bio`：简介
* `profile_version`：频道资料版本号
* `created_at`：频道创建时间
* `updated_at`：资料最后更新时间
* `signature`：owner 对频道资料的签名

---

## 3.2 频道头对象 `ChannelHead`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "owner_peer_id": "12D3KooWxxxx",
  "owner_version": 1,
  "latest_message_id": 200,
  "profile_version": 3,
  "channel_revision": 168,
  "updated_at": 1710009999,
  "signature": "base64sig"
}
```

### 字段说明

* `channel_id`：频道 ID
* `owner_peer_id`：当前 owner
* `owner_version`：owner 版本，v1 固定为 1
* `latest_message_id`：该频道当前最大消息号
* `profile_version`：频道资料版本
* `channel_revision`：频道变更序号
* `updated_at`：head 更新时间
* `signature`：owner 签名

---

## 3.3 消息对象 `ChannelMessage`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "message_id": 15,
  "version": 2,
  "owner_version": 1,
  "creator_peer_id": "12D3KooWxxxx",
  "author_peer_id": "12D3KooWxxxx",
  "created_at": 1710000000,
  "is_deleted": false,
  "content_type": "file",
  "text": "这是更新后的内容",
  "files": [
    {
      "cid": "bafy...",
      "name": "report.pdf",
      "mime": "application/pdf",
      "size": 2481234,
      "sha256": "..."
    }
  ],
  "signature": "base64sig"
}
```

### 字段说明

* `channel_id`：所属频道 UUIDv7
* `message_id`：频道内消息编号，从 1 开始
* `version`：消息当前版本号
* `owner_version`：owner 版本，v1 固定为 1
* `creator_peer_id`：最初创建该消息的 peer_id
* `author_peer_id`：当前这个版本的签名者，v1 必须等于当前频道 owner
* `created_at`：消息首次创建时间，永不改变
* `is_deleted`：是否已删除
* `content_type`：`text` / `file` / `deleted`
* `text`：文本内容
* `files`：附件列表
* `signature`：对当前完整消息状态的签名

---

## 3.4 文件对象 `ChannelFile`

```json
{
  "cid": "bafy...",
  "name": "image.jpg",
  "mime": "image/jpeg",
  "size": 123456,
  "sha256": "abc...",
  "width": 1080,
  "height": 720,
  "duration": 0,
  "thumbnail_cid": "bafy-thumb..."
}
```

### 说明

* 文件独立存储，发布的时候提交到本节点的ipfs
* 消息只保存文件引用
* v1 不做文件加密
* v1 不承诺删除后全网物理擦除文件

---

## 3.5 轻量变更对象 `ChannelChange`

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "channel_revision": 121,
  "change_type": "message",
  "message_id": 201,
  "version": 1,
  "is_deleted": false,
  "profile_version": null,
  "created_at": 1710010000
}
```

或资料变更：

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "channel_revision": 122,
  "change_type": "profile",
  "message_id": null,
  "version": null,
  "is_deleted": null,
  "profile_version": 4,
  "created_at": 1710010100
}
```

### 说明

`ChannelChange` 不是完整事件日志，只用于同步索引。

它只记录：

* 哪个 revision 发生了变化
* 变的是消息还是资料
* 哪条消息的新版本是多少

它不保存：

* 旧正文
* 编辑前内容
* 删除前内容
* 完整事件回放历史

---

# 4. 核心规则

## 4.1 一个用户可以创建多个频道

同一个 `owner_peer_id` 可以拥有多个 `channel_id`：

* `channel_id = uuidv7_1`
* `channel_id = uuidv7_2`
* `channel_id = uuidv7_3`

这些频道彼此独立：

* 各自有独立资料
* 各自有独立消息序号
* 各自有独立 `latest_message_id`
* 各自有独立 `channel_revision`

---

## 4.2 消息编号规则

`message_id` 仅在 **单频道内部** 递增：

* 第 1 条消息：`message_id = 1`
* 第 2 条消息：`message_id = 2`

不同频道内都可以有 `message_id = 1`，不冲突。

消息的逻辑主键是：

```text
(channel_id, message_id)
```

数据库层通过 `channel_db_id + message_id` 实现。

---

## 4.3 版本规则

### 创建消息

* `version = 1`

### 修改消息

* `version = old_version + 1`

### 删除消息

* `version = old_version + 1`

同一条消息始终以 **更大的 `version` 为准**。

---

## 4.4 频道变更序号规则

`channel_revision` 在以下任一操作后加 1：

* 创建消息
* 修改消息
* 删除消息
* 修改频道资料

例如：

* 初始：`channel_revision = 0`
* 发布第一条消息：`channel_revision = 1`
* 编辑消息 1：`channel_revision = 2`
* 删除消息 3：`channel_revision = 3`
* 修改频道简介：`channel_revision = 4`

---

## 4.5 删除规则

删除时不物理删除记录，而是覆盖为墓碑：

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "message_id": 15,
  "version": 3,
  "owner_version": 1,
  "creator_peer_id": "12D3KooWxxxx",
  "author_peer_id": "12D3KooWxxxx",
  "created_at": 1710000000,
  "is_deleted": true,
  "content_type": "deleted",
  "text": "",
  "files": [],
  "signature": "base64sig"
}
```

前端显示：

```text
该消息已删除
```

---

## 4.6 查看排序规则

频道阅读视图固定按：

```text
ORDER BY message_id DESC
```

这意味着：

* 新创建的消息排前面
* 编辑旧消息不会顶到前面
* 删除旧消息不会改变位置

---

# 5. v1 不实现但预留的 owner 转移

## 5.1 v1 结论

**v1 不实现 owner 转移功能。**

也就是：

* 创建频道后，`owner_peer_id` 固定
* `owner_version` 固定为 `1`
* 所有消息都必须由当前 owner 签名
* 暂不支持把频道所有权从 A 转给 B

---

## 5.2 为什么预留

未来可能需要：

* 频道交接
* 身份迁移
* 私钥轮换
* 账号升级

因此 v1 预留以下设计：

### 预留字段

* `ChannelProfile.owner_version`
* `ChannelHead.owner_version`
* `ChannelMessage.owner_version`

v1 中这些字段都固定为：

```text
owner_version = 1
```

---

## 5.3 预留接口（只定义，不实现）

### 发起 owner 转移

```text
POST /channels/{channel_id}/owner-transfer/init
```

### 接受 owner 转移

```text
POST /channels/{channel_id}/owner-transfer/accept
```

### 查询 owner 转移状态

```text
GET /channels/{channel_id}/owner-transfer
```

### 说明

* v1 中这些接口直接返回“未实现”
* v1 数据库不需要真正写 owner 历史逻辑
* v1 只保留字段和接口壳

---

# 6. 频道资料规则

## 6.1 创建频道

创建频道时：

* 生成 `channel_id = UUIDv7`
* 设置 `owner_peer_id`
* 设置 `owner_version = 1`
* 设置 `profile_version = 1`
* 设置 `channel_revision = 0`
* `created_at = now`
* `updated_at = now`
* 对频道资料签名

---

## 6.2 修改频道资料

允许 owner 修改：

* `name`
* `avatar`
* `bio`

修改时：

* `profile_version = old_profile_version + 1`
* `channel_revision = old_channel_revision + 1`
* `updated_at = now`
* `owner_version` 保持 1
* 重新签名
* 追加一条 `ChannelChange(change_type = "profile")`

---

# 7. 消息发布、修改、删除规则

## 7.1 创建消息

创建时：

* `message_id = channel.latest_message_id + 1`
* `version = 1`
* `owner_version = 1`
* `creator_peer_id = owner_peer_id`
* `author_peer_id = owner_peer_id`
* `created_at = now`
* `is_deleted = false`
* 填入 `text/files`
* 生成签名
* `latest_message_id = message_id`
* `channel_revision = old_channel_revision + 1`
* 写入一条 `ChannelChange(change_type = "message")`

---

## 7.2 修改消息

修改时：

* `channel_id` 不变
* `message_id` 不变
* `version = old_version + 1`
* `owner_version = 1`
* `creator_peer_id` 不变
* `author_peer_id = owner_peer_id`
* `created_at` 不变
* `is_deleted = false`
* 更新 `text/files`
* 重新签名
* `channel_revision = old_channel_revision + 1`
* 写入一条 `ChannelChange(change_type = "message")`

---

## 7.3 删除消息

删除时：

* `channel_id` 不变
* `message_id` 不变
* `version = old_version + 1`
* `owner_version = 1`
* `creator_peer_id` 不变
* `author_peer_id = owner_peer_id`
* `created_at` 不变
* `is_deleted = true`
* `content_type = "deleted"`
* `text = ""`
* `files = []`
* 重新签名
* `channel_revision = old_channel_revision + 1`
* 写入一条 `ChannelChange(change_type = "message")`

---

```text
owner_version
name
avatar
bio
profile_version
created_at

```

---

## 8.2 频道头签名字段

签名覆盖字段建议为：

```text
channel_id
owner_peer_id
owner_version
latest_message_id
profile_version
channel_revision
updated_at
```

---

## 8.3 消息签名字段

签名覆盖字段建议为：

```text
channel_id

owner_version
creator_peer_id
author_peer_id
created_at
is_deleted
content_type
text

```

---

## 8.4 验签规则

收到频道资料后：

* `channel_id` 必须是合法 UUIDv7
* `signature` 必须由 `owner_peer_id` 验签通过

收到频道头后：

* `signature` 必须由 `owner_peer_id` 验签通过

收到消息后：

* `channel_id` 必须存在
* `owner_version` 在 v1 必须为 `1`
* `author_peer_id` 必须等于该频道当前 `owner_peer_id`
* `signature` 必须验证通过

# 9. 同步设计

由于不保留事件日志，同步采用：

> **当前状态同步 + 首次只拉最新一页 + 用户按需翻页拉历史 + 订阅后接收后续更新**

---

## 9.1 首次进入频道

首次进入频道时，只拉取：

1. `ChannelProfile`
2. `ChannelHead`
3. 最新一页消息列表

**不拉取全部历史消息。**

这样做的目标是：

* 更快打开频道
* 更省流量
* 更适合消息较多的频道
* 用户只有真的向上翻页时，才拉取更旧历史

---

## 9.2 首次进入频道的推荐流程

### 第 1 步：获取频道资料

* 拉 `ChannelProfile`

### 第 2 步：获取频道头

* 拉 `ChannelHead`
* 记住 `head.channel_revision = R`

### 第 3 步：拉取最新一页消息

例如最新 20 条：

* 200
* 199
* 198
* ...
* 181

### 第 4 步：开始订阅频道

订阅时带上：

```text
since_revision = R
```

### 第 5 步：补齐 `R` 之后的变更

订阅建立后，服务端或客户端应确保：

* `channel_revision > R` 的变更不会漏掉

---

## 9.3 为什么订阅要带 `since_revision`

这是为了避免竞态问题。

例如：

1. 客户端先拿到 `head.channel_revision = 120`
2. 客户端拉完最新一页消息
3. 这时频道又发生了新变更，变成 revision 121
4. 客户端再开始订阅

如果订阅不带 `since_revision`，就可能漏掉 revision 121。

因此订阅必须支持：

```text
since_revision = 120
```

---

## 9.4 用户查看历史消息

用户向上翻页时，才继续拉更老的消息。

### 规则

* 按 `before_message_id` 拉更老消息
* 仍然按 `message_id DESC` 返回

例如：

已有 200~181，用户上滑后继续拉：

* 180
* 179
* 178
* ...
* 161

---

## 9.5 订阅后的增量同步

客户端开始订阅后，不再重新拉全部消息，只接收“之后发生的更新”。

订阅推送内容可以是：

* 新消息创建
* 旧消息被修改
* 旧消息被删除
* 频道资料被更新

推荐通过 `channel_changes` 推送变更索引，再按需拉具体对象。

---

## 9.6 增量同步不要求补全全部历史

订阅后的目标不是补全整个频道历史，而是维护：

* 当前已加载区域
* 之后发生的新变化

也就是说：

* 首次只拿最新一页
* 之后通过订阅获得新的更新
* 更老历史只有在用户主动翻页时才加载

---

## 9.7 本地状态建议

客户端本地建议维护：

### 频道级状态

* `latest_known_revision`
* `latest_loaded_message_id`
* `oldest_loaded_message_id`

### 含义

* `latest_known_revision`：当前客户端已同步到的频道 revision
* `latest_loaded_message_id`：当前已加载消息中最大的 message_id
* `oldest_loaded_message_id`：当前已加载消息中最小的 message_id

用途：

* `latest_known_revision` 用于订阅续传
* `oldest_loaded_message_id` 用于向上翻页

---

# 10. `channel_changes` 分页规则

`channel_changes` 必须支持按 `channel_revision` 增量分页拉取，不能假设一次拉完。

## 10.1 接口参数

* `since_revision`
* `limit`

表示：

* 返回 `channel_revision > since_revision` 的变更
* 最多返回 `limit` 条
* 按 `channel_revision ASC` 排序

---

## 10.2 返回格式建议

```json
{
  "channel_id": "0195f3f0-8d4a-7c12-b2c1-9db1f0a9e123",
  "current_revision": 168,
  "has_more": true,
  "next_since_revision": 220,
  "items": [
    {
      "channel_revision": 121,
      "change_type": "message",
      "message_id": 201,
      "version": 1,
      "is_deleted": false
    },
    {
      "channel_revision": 122,
      "change_type": "message",
      "message_id": 195,
      "version": 2,
      "is_deleted": false
    }
  ]
}
```

---

## 10.3 客户端处理建议

处理一批 changes 时，客户端应先按 `message_id` 去重，只保留同一批中的最新 version，再去拉正文，避免重复拉取同一条消息。

---

# 11. libp2p 通信与同步设计

## 11.1 总体原则

频道系统底层使用 **libp2p** 作为节点之间的通信、发现和同步网络。

libp2p 在本系统中承担三类职责：

1. **实时推送**

   * 订阅频道的节点通过 PubSub 接收该频道的最新更新通知

2. **按需拉取**

   * 节点可通过 libp2p stream protocol 从 owner 或其它订阅节点拉取频道资料、消息、历史和变更索引

3. **节点发现**

   * 订阅某频道的节点可发现其它也订阅该频道的节点，并从它们拉数据

## 11.2 角色定义

* 维护最新 `channel_revision`

负责：

* 订阅频道 PubSub topic
* 接收频道最新更新推送

* 可作为只读副本向其它节点提供拉取服务

---

未订阅某频道时：

* 不参与该频道实时 PubSub
* 可以按需临时拉频道数据

PubSub 只负责：

* 推送最新更新
* 推送轻量变更通知
* 帮助在线订阅节点快速感知新变更

PubSub **不负责完整历史同步**。

---

## 11.4 Stream Protocol 用于按需拉取

libp2p stream protocol 负责：

* 拉频道资料
* 拉频道头
* 拉最新一页消息
* 拉更老历史消息
* 拉单条消息
* 拉 change index
* 拉 owner 名下的频道列表（可选）

也就是：

> 实时靠 PubSub，历史和精确对象靠 stream pull。

## 11.5 订阅节点之间的数据拉取

* 获取单条消息
* 获取 changes

但它只能返回自己本地实际持有的数据。

节点可以只缓存：

* 最新一页

```text id="4o7ztg"

```

* 客户端调用路径统一
* 更适合 v1 快速实现
* 更适合 AI 生成代码

## 13.5 单一 stream RPC 的使用方式

v1 推荐采用：

> **一次请求对应一个 stream**

也就是：

1. 打开一个 stream
2. 发送一个 RPC 请求
3. 接收一个 RPC 响应
4. 关闭该 stream

v1 **不做**单个长连接 stream 上的多请求复用。

这样实现更简单，错误隔离更清晰。

# 17. 节点发现与频道数据源发现

## 17.1 订阅节点发现

订阅某频道的节点之间，需要能互相发现。

推荐方式：

### 方式 A：PubSub mesh 内自然感知

加入频道 topic 后，可获得部分在线邻居。

### 方式 B：按 `channel_id` 做 provider advertisement

节点声明：

> 我可为这个 `channel_id` 提供拉取服务

这样其它节点可以通过 DHT / provider 查找到可拉取的节点。

---

## 17.2 推荐发现后的行为

客户端查到某频道可用节点后，应通过统一 RPC 协议发起请求，而不是依赖不同协议名。

也就是说：

* 发现 peer
* 打开 `/app/channel/rpc/1.0.0`
* 发送 `method + body`
* 获取响应

# 18. 给 AI 的实现要求（单一 stream protocol 版）

请按以下规则实现 libp2p 通信层：

1. 使用 libp2p 作为节点间通信层

   * `/app/channel/rpc/1.0.0`
6. 统一 RPC 协议通过 `method` 字段区分不同动作
7. v1 推荐“一次请求一个 stream”的模式，不做长连接多路复用

9. owner transfer 相关方法先保留，但返回 `not_implemented`
10. 订阅节点应可作为只读副本，通过同一个 RPC 协议向其它节点提供自己已有的数据

# 15. owner 设备不持有全量内容时的规则

协议必须允许：

> owner 的某个设备在不持有频道全量内容时，仍然可以发新消息或更新指定消息。

---

## 15.1 发布新消息

owner 设备至少需要知道：

* 当前 `latest_message_id`
* 当前 `channel_revision`

然后才能生成：

* 新 `message_id`
* 新 `channel_revision`

---

## 15.2 修改旧消息

owner 设备修改旧消息前，必须先拉取目标消息当前版本：

* 确认当前 `version`
* 确认当前 `is_deleted`
* 确认当前可编辑状态

然后再生成：

* `version = old_version + 1`
* `channel_revision = old_channel_revision + 1`

也就是：

> 不能盲改，必须先拿目标消息最新状态。

```sql
  UNIQUE KEY uk_channel_revision (channel_db_id, channel_revision),
  KEY idx_channel_changes_lookup (channel_db_id, channel_revision),
  CONSTRAINT fk_channel_changes_channel
    FOREIGN KEY (channel_db_id) REFERENCES channels(id)
    ON DELETE CASCADE
);
```

### 说明

* `channel_changes` 只是同步索引，不是完整事件正文存档
* 可以分页按 `channel_revision ASC` 拉取

---

## 17.4 预留：`channel_owner_transfers`

v1 不实现，可不建表。
如果想提前占位，也可以建空表，但不是必须。

# 18. 查询规则

## 18.1 查看频道消息

按 `message_id DESC`：

```sql
SELECT *
FROM channel_messages
WHERE channel_db_id = ?
ORDER BY message_id DESC
LIMIT ?
```

---

## 18.2 向上翻页

```sql
SELECT *
FROM channel_messages
WHERE channel_db_id = ?
  AND message_id < :before_message_id
ORDER BY message_id DESC
LIMIT ?
```

不使用 `OFFSET`。

---

## 18.3 查询 changes

```sql
SELECT *
FROM channel_changes
WHERE channel_db_id = ?
  AND channel_revision > :since_revision
ORDER BY channel_revision ASC
LIMIT ?
```

---

## 18.4 查询 owner 名下频道

```sql
SELECT *
FROM channels
WHERE owner_peer_id = ?
ORDER BY created_at DESC
LIMIT ?
```

# 19. 业务校验规则

## 19.1 创建频道

* `channel_id` 必须是合法 UUIDv7
* `owner_version = 1`
* `profile_version = 1`
* `channel_revision = 0`

---

## 19.2 发布消息

* 只有当前频道 owner 可以发布
* 新消息的 `message_id` 必须等于 `latest_message_id + 1`
* `version = 1`
* `owner_version = 1`

---

## 19.3 修改消息

* 只有当前频道 owner 可以修改
* 目标消息必须存在
* 新版本必须满足 `version > old_version`
* `created_at` 不可变
* `creator_peer_id` 不可变

---

## 19.4 删除消息

* 只有当前频道 owner 可以删除
* 目标消息必须存在
* `version > old_version`
* 覆盖为墓碑

---

## 19.5 同步覆盖规则

对于本地已有消息：

* 如果远端 `version > local.version`，则覆盖本地
* 如果 `version <= local.version`，则丢弃

---

## 19.6 changes 的可信性

`channel_changes` 本身只是同步提示，不是最终可信对象。

最终仍以：

* `ChannelProfile`
* `ChannelHead`
* `ChannelMessage`

的签名验真为准。

也就是说：

> changes 负责告诉客户端“去拉什么”，真正可信的是被拉回来的已签名对象。

---

# 20. v1 明确不做的内容

以下内容不纳入 v1：

* owner 转移
* 多管理员
* 管理员权限分级
* 多写者频道
* 评论
* 点赞
* 完整编辑历史
* 完整删除历史
* 私密频道
* 附件加密
* 强制全网物理删除

# 22. 给 AI 的最短实现要求

请按以下规则实现 v1：

1. 频道对外唯一标识使用 `UUIDv7`
2. 一个 `owner_peer_id` 可以创建多个频道
3. 每个频道只有一个 owner
4. v1 不支持 owner 转移，但保留：

   * `owner_version` 字段
   * owner transfer 接口路由
   * 接口返回未实现
5. 本地数据库 `channels` 使用整数主键 `id`
6. `channel_messages` 通过 `channel_db_id` 关联 `channels.id`
7. 另建 `channel_changes` 表，用于轻量同步索引
8. 消息按频道内 `message_id` 从 1 自增
9. 频道查看按 `message_id DESC`
10. 消息采用当前状态覆盖模型，不保留完整事件日志
11. 消息新旧只看 `version`
12. 删除采用墓碑覆盖
13. 文件独立存储，消息只保存文件引用
14. 首次进入频道只拉最新一页消息，不拉全量历史
15. 用户向上翻页时，再拉更老历史
16. 订阅频道后，只接收并补齐之后的更新
17. `channel_changes` 必须支持按 `since_revision + limit` 分页拉取
18. 订阅底层使用 libp2p PubSub
19. 按需拉取资料、消息、changes 使用 libp2p stream protocol
20. 订阅节点应可作为只读副本向其它节点提供自己已有的数据
21. 节点应支持按 `channel_id` 发现可提供该频道数据的其它订阅节点
22. owner 设备不要求持有频道全部内容，但修改旧消息前必须先拉该消息最新版本
23. 所有频道资料、频道头和消息都必须签名并验签
24. `channel_changes` 不是最终可信对象，最终可信以 owner 签名对象为准
