**实现说明书**

项目目标：为现有 `Go + libp2p` 去中心化聊天系统增加“中继协助连接 + 端到端加密隧道”能力。  
要求：当 `A` 无法直连 `B` 时，通过中继 `R` 建立可复用会话；建立后后续通信优先走该会话；中继不可见通信内容；会话存活期间每 `5-15s` 发送一次心跳。

**一、强约束**

1. 不修改现有业务语义：聊天、离线消息、ACK 的语义保持不变。
2. 中继节点只能看到路由控制信息，不能看到业务明文。
3. 必须端到端加密，密钥只在 `A/B` 两端生成和持有。
4. 必须使用 libp2p stream protocol，不允许引入 HTTP。
5. 必须支持会话复用，不能每条消息都重新建链。
6. 必须支持心跳保活、故障检测、重连和换中继。
7. 必须防止重放、伪造握手、伪造心跳、伪造数据包。
8. V1 只做单中继：`A -> R -> B`。
9. V1 不做多跳路由、不做匿名网络、不做复杂信誉系统。
10. V1 中 `device_id` 不参与中继设计。

**二、角色定义**

1. `A`：源节点，发起连接与数据发送。
2. `B`：目标节点，接受连接与数据接收。
3. `R`：中继节点，只转发控制面和密文数据面，不解密业务内容。

**三、协议列表**

新增协议：

1. `/chat/relay/connect/1.0.0`
   用于中继会话建立。
2. `/chat/relay/handshake/1.0.0`
   用于端到端临时密钥协商。
3. `/chat/relay/data/1.0.0`
   用于转发端到端加密后的业务数据。
4. `/chat/relay/heartbeat/1.0.0`
   用于 relay session 保活。

现有协议：

1. `/chat/offline/store/1.0.0`
2. `/chat/offline/fetch/1.0.0`
3. `/chat/offline/ack/1.0.0`

要求：现有业务协议帧必须作为“被加密后的内层 payload”通过 `/chat/relay/data/1.0.0` 传输。

**四、连接建立流程**

1. `A` 先尝试直连 `B`。
2. 若直连失败，`A` 进入 relay 候选选择。
3. `A` 选择一个可达中继 `R`，发送 `RelayConnectRequest`。
4. `R` 将 connect 请求转发给 `B`。
5. `B` 校验请求、决定是否接受，返回 `RelayConnectResponse`。
6. `A/B` 在 relay session 已建立的前提下，通过 `/chat/relay/handshake/1.0.0` 交换临时公钥。
7. `A/B` 基于临时密钥协商出 `K_session`。
8. 会话进入 `RelayConnected`，后续业务优先走该 session。
9. 业务层消息先序列化原始业务帧，再用 `K_session` 加密后通过 `/chat/relay/data/1.0.0` 转发。

**五、连接状态机**

每个目标 `peer_id` 维护状态：

1. `DirectConnected`
2. `DirectDialFailed`
3. `RelayNegotiating`
4. `RelayHandshake`
5. `RelayConnected`
6. `RelayDegraded`
7. `Disconnected`

迁移规则：

1. 直连成功进入 `DirectConnected`。
2. 直连失败进入 `DirectDialFailed`。
3. 选中 relay 后进入 `RelayNegotiating`。
4. connect 成功后进入 `RelayHandshake`。
5. E2E 握手成功后进入 `RelayConnected`。
6. 心跳连续失败进入 `RelayDegraded`。
7. 超过阈值则进入 `Disconnected` 并重新选路。
8. 若后台探测到直连恢复，可从 `RelayConnected` 平滑切回 `DirectConnected`。

**六、消息结构**

`RelayConnectRequest`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "A",
  "dst_id": "B",
  "timestamp": 1710000000,
  "ttl_sec": 60,
  "signature": {
    "algorithm": "ed25519",
    "value": "base64"
  }
}
```

签名覆盖：

`version, session_id, src_id, dst_id, timestamp, ttl_sec`

`RelayConnectResponse`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "B",
  "dst_id": "A",
  "relay_id": "R",
  "accepted": true,
  "expire_at": 1710000060,
  "signature": {
    "algorithm": "ed25519",
    "value": "base64"
  }
}
```

签名覆盖：

`version, session_id, src_id, dst_id, relay_id, accepted, expire_at`

`RelayHandshakeRequest`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "A",
  "dst_id": "B",
  "eph_pub": "base64",
  "cipher_suite": "x25519-chacha20poly1305",
  "timestamp": 1710000001,
  "signature": {
    "algorithm": "ed25519",
    "value": "base64"
  }
}
```

`RelayHandshakeResponse`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "B",
  "dst_id": "A",
  "eph_pub": "base64",
  "cipher_suite": "x25519-chacha20poly1305",
  "timestamp": 1710000002,
  "signature": {
    "algorithm": "ed25519",
    "value": "base64"
  }
}
```

签名覆盖：

`version, session_id, src_id, dst_id, eph_pub, cipher_suite, timestamp`

`RelayDataFrame`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "A",
  "dst_id": "B",
  "packet_seq": 12,
  "cipher": {
    "algorithm": "chacha20-poly1305",
    "nonce": "base64",
    "ciphertext": "base64"
  }
}
```

说明：

`ciphertext` 内部是加密后的原始业务帧字节，不是明文 JSON。

`RelayHeartbeat`

```json
{
  "version": 1,
  "session_id": "uuid",
  "src_id": "A",
  "dst_id": "B",
  "relay_id": "R",
  "ping_id": "uuid",
  "sent_at": 1710000010,
  "signature": {
    "algorithm": "ed25519",
    "value": "base64"
  }
}
```

签名覆盖：

`version, session_id, src_id, dst_id, relay_id, ping_id, sent_at`

**七、密码与密钥要求**

1. 身份签名：`ed25519`
2. 临时密钥协商：`x25519`
3. 数据加密：`chacha20-poly1305`
4. `K_session` 必须绑定 `session_id`
5. 每个数据包使用唯一 nonce
6. 重建 session 后必须派生新密钥，禁止复用旧会话密钥
7. relay 不参与会话密钥计算

**八、路由与复用规则**

本地维护 `RouteEntry`：

```json
{
  "peer_id": "B",
  "route_type": "relay",
  "relay_id": "R",
  "session_id": "uuid",
  "state": "RelayConnected",
  "rtt_ms": 120,
  "last_heartbeat_at": 1710000100,
  "fail_count": 0,
  "expire_at": 1710003600
}
```

优先级：

1. `DirectConnected`
2. `RelayConnected`
3. 否则重新探测和建链

要求：

1. 对同一个 `dst_id` 最多只有一个“首选路由”。
2. 允许保留一个备用 relay 路由，但 V1 不要求同时活跃。
3. 后续所有发往 `B` 的业务消息优先复用已有 `session_id`。

**九、心跳规则**

1. 心跳间隔每次随机在 `5-15s`。
2. 单次心跳超时建议 `10s`。
3. 连续 `3` 次失败：标记 `RelayDegraded`。
4. 连续 `5` 次失败：标记 `Disconnected`。
5. 心跳成功时更新 `last_heartbeat_at`、`rtt_ms`。
6. `RelayConnected` 和 `RelayDegraded` 状态都应允许后台直连恢复探测。

**十、候选中继选择**

V1 候选来源可包括：

1. 已连接节点
2. 历史成功转发过的节点
3. 明确配置的 bootstrap 或 relay 节点

排序建议：

1. 当前可达
2. 最近成功
3. RTT 更低
4. 当前负载更低
5. 非黑名单

V1 不要求复杂全网发现，只要支持本地候选列表即可。

**十一、中继节点行为要求**

1. `R` 必须校验 connect 请求格式和签名是否完整。
2. `R` 只转发，不解密 `RelayDataFrame.ciphertext`。
3. `R` 可拒绝超载、超频、无效、过期请求。
4. `R` 必须按 `session_id` 维护临时转发表。
5. `R` 必须限制每个源节点的建链频率、会话数、带宽、心跳速率。
6. `R` 必须支持 session 超时回收。
7. `R` 不应持久化业务密文正文，最多做短暂内存转发缓存。

**十二、目标节点行为要求**

1. `B` 必须校验 connect 请求中 `dst_id == B`。
2. `B` 必须校验 `A` 的 connect 签名。
3. `B` 必须拒绝过期的 connect 请求。
4. `B` 必须防止 `session_id` 重放。
5. `B` 必须在 handshake 中校验 `A` 的身份签名。
6. `B` 解密 `RelayDataFrame` 后，再按内层协议分发到原有业务处理器。
7. `B` 不能因为 relay 存在而跳过上层业务验签或权限校验。

**十三、错误码**

新增错误码：

1. `DIRECT_DIAL_FAILED`
2. `RELAY_NOT_AVAILABLE`
3. `RELAY_CONNECT_REJECTED`
4. `RELAY_CONNECT_TIMEOUT`
5. `RELAY_UNAUTHORIZED`
6. `RELAY_RATE_LIMITED`
7. `SESSION_NOT_FOUND`
8. `SESSION_EXPIRED`
9. `SESSION_REPLAYED`
10. `HANDSHAKE_INVALID`
11. `HANDSHAKE_TIMEOUT`
12. `HEARTBEAT_TIMEOUT`
13. `HEARTBEAT_INVALID`
14. `RELAY_PAYLOAD_TOO_LARGE`
15. `ROUTE_NOT_AVAILABLE`

**十四、实现边界**

必须实现：

1. 单中继 connect
2. E2E 握手
3. relay data 转发
4. 5-15 秒随机心跳
5. session 复用
6. 心跳失败重连
7. 直连恢复切换

暂不实现：

1. 多跳 relay
2. 流量混淆
3. 复杂信誉系统
4. relay 经济模型
5. 群组多播 relay

**十五、测试验收标准**

必须覆盖以下场景：

1. `A` 无法直连 `B` 时，可通过 `R` 建立 session。
2. session 建立后，后续业务消息不再重复建链。
3. relay 无法看到业务明文内容。
4. 篡改 handshake 公钥会被签名校验拒绝。
5. 篡改心跳包会被拒绝。
6. 重放旧 `session_id` 会被拒绝。
7. 连续心跳失败后 route 状态会降级并触发重连。
8. 直连恢复后能从 relay 平滑切回直连。
9. relay 超载时会返回明确错误。
10. 业务层原有 `/chat/offline/store|fetch|ack` 通过 relay 密文通道仍可正常工作。

**十六、对 Cursor/Codex 的开发要求**

1. 先补协议类型定义、状态结构、路由表结构。
2. 再实现 connect、handshake、heartbeat 三条控制协议。
3. 再实现 relay data 封装与内层业务帧加解密。
4. 最后接入现有连接选择逻辑，让后续通信优先走 `RelayConnected`。
5. 不要重构现有业务协议处理器，只在其外层增加 relay transport 层。
6. 所有新增协议继续使用 length-prefixed JSON。
7. 所有 stream 必须有读写 deadline 和 handler timeout。
8. 所有关键状态变更必须可观测：日志、错误码、测试。
