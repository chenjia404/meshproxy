# 《去中心化匿名代理网络协议与本地 SOCKS5 Agent 设计文档》

## 1. 文档信息

**文档名称**：去中心化匿名代理网络协议与本地 SOCKS5 Agent 设计文档
**文档版本**：v1.0
**目标读者**：协议设计者、Go 后端开发者、P2P 网络开发者、客户端开发者、AI Codegen 系统
**实现建议语言**：Go
**推荐底层网络库**：go-libp2p
**推荐运行平台**：Linux / Windows / macOS / Android

---

## 2. 项目概述

本项目目标是设计一套**去中心化匿名代理网络协议**，网络中的节点分为两类能力角色：

* **Relay（中继节点）**
* **Exit（出口节点）**

其中：

* 所有节点默认都运行为 **Relay**
* 部分节点可额外启用 **Exit** 能力
* 所有节点在本地都必须启动一个 **SOCKS5 Agent**
* 本地应用通过 SOCKS5 接入匿名代理网络
* 用户流量可经由多跳 Relay 后通过 Exit 节点访问公网
* Relay 节点仅知道上一跳与下一跳，不知道目标明文地址，转发内容为分层加密
* 节点通过去中心化方式进行发现、广播和加入网络

本系统可视为：

**去中心化匿名代理覆盖网络（Overlay Network） + 本地 SOCKS5 接入代理（Local Agent）**

---

## 3. 设计目标

### 3.1 核心目标

1. 节点去中心化发现
2. 支持多跳匿名转发
3. Relay 节点只能看到局部路径
4. Exit 节点负责最终公网访问
5. 所有节点本地自动提供 SOCKS5 代理
6. 支持直连 Exit 与多跳匿名两种路径模式
7. 支持后续扩展计费、信誉、限流、黑名单、策略路由

### 3.2 非目标

v1 不追求：

* 抵御国家级全球流量分析
* 完整混淆流量指纹
* 完整 UDP / QUIC 出口生态
* 完整去信任计费结算
* 完整匿名支付系统
* 极限高性能优于传统中心化代理

---

## 4. 术语定义

### 4.1 节点

运行本协议的软件实例，具有唯一身份、公钥、网络地址和能力描述。

### 4.2 Relay

中继节点。仅负责接收、解一层包头并转发到下一跳，不访问公网目标。

### 4.3 Exit

出口节点。作为最后一跳，解最后一层 onion 消息并访问公网目标。

### 4.4 Circuit

匿名电路。由客户端动态构建的一条多跳路径，例如：

`Local Node -> Relay A -> Relay B -> Exit X`

### 4.5 Stream

电路中的逻辑流。一个本地 SOCKS5 连接通常对应一个 Stream。

### 4.6 Local Agent

节点本地运行的接入层代理组件，默认提供 SOCKS5 服务。

### 4.7 Node Descriptor

节点对外广播的能力描述信息，包括 PeerId、地址、角色、带宽、出口国家、TTL 等。

### 4.8 Hop

匿名路径中的一个远程节点。**本地节点不是匿名 hop，仅是入口和电路构建者。**

## 5. 系统总体架构

### 5.1 总体拓扑

```text
+----------------------+
| Local Apps / Browser |
+----------+-----------+
           |
           v
+----------------------+
| Local SOCKS5 Agent   |
| 127.0.0.1:1080       |
+----------+-----------+
           |
           v
+----------------------+
| Local Node           |
| - Path Selector      |
| - Circuit Manager    |
| - Stream Manager     |
| - Discovery Manager  |
| - Relay Service      |
+----------+-----------+
           |
           v
      P2P Overlay Network
           |
   +-------+-------+
   |               |
   v               v
+--------+     +--------+
| Relay  | --> | Relay  |
+--------+     +--------+
                   |
                   v
               +--------+
               | Exit   |
               +--------+
                   |
                   v
                Internet
```

### 5.2 核心结构分层

系统拆分为以下逻辑层：

1. **Identity Layer**：身份与密钥
2. **Discovery Layer**：节点发现与广播
3. **Transport Layer**：节点间安全传输
4. **Circuit Layer**：匿名电路建立
5. **Relay Layer**：多跳中继转发
6. **Exit Layer**：公网访问出口
7. **Local Agent Layer**：本地 SOCKS5 接入
8. **Policy & Reputation Layer**：策略与评分

## 6. 节点角色与运行模式

## 6.1 角色定义

协议层只有两种核心能力角色：

* `relay`
* `exit`

其中：

* 所有节点默认至少具备 `relay`
* 只有启用出口功能时，节点才额外具备 `exit`

### 6.2 运行模式

本系统**没有 client-only 模式**。所有节点启动后默认就是 Relay，并在本地启动 SOCKS5。

支持以下运行模式：

#### 模式 A：relay（默认）

功能：

* 本地启动 SOCKS5
* 加入 P2P 网络
* 发布 relay 能力
* 接受并转发其他用户电路
* 自己作为本地入口构建电路访问远程 exit

#### 模式 B：relay+exit

功能：

* 具备 relay 全部能力
* 额外发布 exit 能力
* 可作为电路最后一跳访问公网
* 解析目标地址并建立外部连接

### 6.3 启动规则

节点程序启动后必须：

1. 加载或生成身份密钥
2. 启动本地 SOCKS5
3. 初始化 P2P Host
4. 加入发现网络
5. 默认发布 relay descriptor
6. 若启用出口，则再发布 exit descriptor
7. 监听匿名电路协议

### 6.4 强制规则

1. 本地 SOCKS5 启动失败，节点启动失败
2. relay 能力默认启用，不可关闭
3. exit 能力默认关闭，需要显式开启
4. 本地请求默认优先走匿名路径，而不是直连出口

## 7. 身份与密钥体系

### 7.1 长期身份密钥

每个节点生成一个长期身份密钥对：

* 算法：`Ed25519`
* 用途：

  * 标识节点身份
  * 派生 PeerId
  * 签名 Node Descriptor
  * 建立节点信誉绑定关系

### 7.2 PeerId

建议直接使用 libp2p PeerId 机制：

* 节点公钥 -> PeerId
* PeerId 作为网络唯一标识

### 7.3 临时会话密钥

为防止长期密钥泄漏影响历史流量，电路建立时每个 hop 还需使用临时密钥协商：

* 算法建议：`X25519`
* 每条 hop 独立协商一组转发会话密钥
* 电路重建时重新协商

### 7.4 密钥分类

建议每 hop 派生以下密钥：

* `k_fwd`：正向流量加密
* `k_bwd`：反向流量加密
* `k_meta`：控制消息加密
* `k_replay`：重放保护相关状态

## 8. 节点发现与广播协议

## 8.1 设计要求

节点发现必须满足：

* 不依赖单中心服务器
* 新节点可快速加入
* 节点上线下线状态可传播
* relay 与 exit 能力可被检索
* descriptor 支持过期与刷新

## 8.2 发现机制组合

建议采用三层机制：

### A. Bootstrap Peers

少量公开引导节点，作用仅为帮助新节点冷启动接入。

### B. DHT

作为主要去中心化发现索引层，存储节点 descriptor 摘要或引用。

### C. Gossip / PubSub

用于快速传播上线、更新、撤销事件。

---

## 8.3 Node Descriptor

每个节点周期性发布签名的能力描述：

```json
{
  "version": 1,
  "peer_id": "12D3KooW...",
  "pubkey": "...",
  "roles": ["relay"],
  "listen_addrs": [
    "/ip4/1.2.3.4/udp/4001/quic-v1/p2p/12D3KooW..."
  ],
  "relay": {
    "max_circuits": 2000,
    "max_bandwidth_mbps": 100,
    "supports_hop": true
  },
  "exit": null,
  "pricing": {
    "mode": "free"
  },
  "expires_at": 1770000000,
  "signature": "..."
}
```

启用出口的节点：

```json
{
  "version": 1,
  "peer_id": "12D3KooW...",
  "pubkey": "...",
  "roles": ["relay", "exit"],
  "listen_addrs": [
    "/ip4/8.8.8.8/udp/4001/quic-v1/p2p/12D3KooW..."
  ],
  "relay": {
    "max_circuits": 5000,
    "max_bandwidth_mbps": 300,
    "supports_hop": true
  },
  "exit": {
    "allow_tcp": true,
    "allow_udp": false,
    "allowed_ports": [80, 443],
    "country": "SG",
    "remote_dns": true
  },
  "pricing": {
    "mode": "free"
  },
  "expires_at": 1770000000,
  "signature": "..."
}
```

## 8.4 DHT Key 设计

建议将节点信息按角色和区域分类索引：

```text
/meshproxy/nodes/relay/<region>/<peer_id>
/meshproxy/nodes/exit/<region>/<peer_id>
/meshproxy/nodes/all/<peer_id>
```

DHT 中不建议存太大的完整描述，可存：

* descriptor 哈希
* 简要元信息
* 取回地址或 CID

---

## 8.5 广播主题

建议 PubSub topics：

* `meshproxy.node.announce.relay`
* `meshproxy.node.announce.exit`
* `meshproxy.node.revoke`
* `meshproxy.node.update`

---

## 8.6 节点上线流程

1. 启动节点
2. 连接 bootstrap peers
3. 完成 identify
4. 初始化 DHT
5. 发布 relay descriptor
6. 若启用 exit，再发布 exit descriptor
7. 订阅 gossip 主题
8. 周期性刷新 descriptor

---

## 8.7 节点下线流程

节点优雅退出时：

1. 广播 revoke 事件
2. 停止接受新 circuit
3. 等待现有 circuit 超时或主动销毁
4. 关闭监听和本地服务

## 9. 传输层协议设计

## 9.1 目标

传输层负责节点之间的可靠加密通信，要求：

* 点对点安全
* 支持多路复用
* 弱网场景可用
* 移动端兼容
* 易于与匿名电路层结合

## 9.2 推荐方案

建议基于 libp2p：

* 传输：`QUIC`
* 安全层：`Noise` 或 TLS
* 多路复用：libp2p stream

### 理由

* QUIC 对移动网络和 NAT 更友好
* libp2p 已支持身份、地址、连接管理和安全握手
* 可将匿名电路协议封装在 libp2p protocol stream 上

## 9.3 地址表示

建议使用 multiaddr：

```text
/ip4/1.2.3.4/udp/4001/quic-v1/p2p/12D3KooW...
```

## 10. 匿名电路设计

## 10.1 目标

实现类似 onion routing 的多跳匿名转发，使每个中继节点只掌握局部信息。

## 10.2 路径模型

典型路径：

```text
Local Node -> Relay A -> Relay B -> Exit X
```

或低延迟路径：

```text
Local Node -> Relay A -> Exit X
```

或直连出口：

```text
Local Node -> Exit X
```

### 关键原则

* 本地节点不算匿名 hop
* relay 仅知道上一跳和下一跳
* exit 知道目标地址，但不应知道完整路径
* 单一 relay 不应知道完整电路

---

## 10.3 Circuit 标识

建议：

* `circuit_id`: 64 位随机数
* 由客户端本地生成
* 在不同 hop 可使用派生后的局部 circuit id 映射，减少全局关联性

---

## 10.4 建链方式

建议采用**逐跳扩展**，而不是一次性提交完整路径。

### Step 1：建立到第一跳 Relay A

客户端发送：

```json
{
  "type": "CREATE",
  "circuit_id": 1001,
  "client_ephemeral_pub": "..."
}
```

Relay A 返回：

```json
{
  "type": "CREATED",
  "circuit_id": 1001,
  "relay_ephemeral_pub": "...",
  "features": {...}
}
```

双方派生 hop key。

### Step 2：通过 A 扩展到 B

客户端向 A 发送：

```json
{
  "type": "EXTEND",
  "circuit_id": 1001,
  "next_peer_id": "B",
  "payload_for_next_hop": "..."
}
```

A 仅知道下一跳是 B，并原样转发扩展载荷。

### Step 3：重复扩展直到 Exit

最后通过前一跳扩展到 Exit X。

---

## 10.5 为什么采用逐跳扩展

优点：

* 第一跳不知道完整路径
* 每跳可独立协商密钥
* 建链失败可局部回退
* 更容易支持多跳和路径重构

---

## 11. Onion 加密模型

## 11.1 分层加密

客户端在发送数据时，对每个 hop 逐层加密：

```text
payload0 = app_data
payload1 = ENC(K_exit, cmd + payload0)
payload2 = ENC(K_relayB, hdrB + payload1)
payload3 = ENC(K_relayA, hdrA + payload2)
send(payload3)
```

每个 relay：

1. 用自己的 hop key 解一层
2. 取出局部头部
3. 获取下一跳信息
4. 将剩余密文转发出去

---

## 11.2 加密算法建议

建议使用 AEAD：

* `ChaCha20-Poly1305`
* 或 `AES-256-GCM`

字段建议：

* `nonce = direction || counter`
* `aad = circuit_id || stream_id || epoch`

---

## 11.3 固定大小 Cell

建议以固定长度 cell 承载 relay 消息，例如：

* 1024 字节
* 或 2048 字节

优点：

* 减少长度侧信道
* 简化分片与复用
* 提高转发统一性

---

## 11.4 重放保护

每 hop 必须维护：

* 单方向计数器
* 重放窗口
* epoch / key version

对于已见过的 nonce / counter 必须拒绝。

---

## 12. Relay 层协议

## 12.1 Relay 职责

Relay 节点必须：

* 接收 CREATE / EXTEND / DATA 等消息
* 解一层 onion
* 仅根据局部头部转发到下一跳
* 不知道最终目标
* 不做 DNS 解析
* 不建立公网连接

## 12.2 Relay 本地状态

每条 circuit 至少维护：

* `local_circuit_id`
* `prev_peer_id`
* `next_peer_id`
* `hop_keys`
* `created_at`
* `last_active_at`
* `stream_count`
* `traffic_stats`

## 12.3 转发规则

Relay 收到 cell 后：

1. 校验消息完整性
2. 依据电路状态定位 hop key
3. 解一层
4. 校验 replay
5. 解析本地控制头
6. 转发剩余密文到下一跳
7. 更新流量统计

## 12.4 Relay 不可做的事

* 读取目标域名或端口
* 修改明文内容
* 强制解析应用层协议
* 记录完整路径

---

## 13. Exit 层协议

## 13.1 Exit 职责

Exit 节点是最后一跳，负责：

* 解最后一层 onion
* 获取目标地址
* 建立公网连接
* 读取返回数据
* 重新封装回电路

## 13.2 Exit 接收的典型命令

* `BEGIN_TCP`
* `BEGIN_UDP`（后续扩展）
* `DNS_RESOLVE`
* `DATA`
* `END`

## 13.3 BEGIN_TCP 消息示例

```json
{
  "type": "BEGIN_TCP",
  "stream_id": 12,
  "target_host": "example.com",
  "target_port": 443,
  "dns_mode": "remote"
}
```

## 13.4 Exit 处理流程

1. 检查出口策略
2. 若目标是域名，则执行远程 DNS
3. 连接目标主机
4. 成功则返回 `CONNECTED`
5. 建立本地 stream 与远端连接之间的双向转发

## 13.5 DNS 策略

默认必须使用 **remote DNS**：

* 域名由 Exit 节点解析
* 本地节点不得自行解析
* 避免本地 DNS 泄漏

## 13.6 出口策略

Exit 节点可配置：

* 允许的协议：TCP / UDP
* 允许的端口白名单
* 国家/地区标签
* 黑名单地址
* 每用户流量限制
* 并发上限

---

## 14. 本地 SOCKS5 Agent 设计

## 14.1 目标

所有节点必须在本地提供标准 SOCKS5 代理，使本机应用无需理解 P2P 协议即可接入匿名网络。

监听地址建议：

* 默认：`127.0.0.1:1080`

## 14.2 定位

SOCKS5 只用于：

* 本地应用 ↔ 本地节点

而不是：

* 节点之间的网络协议

也就是说：

* 本地入口协议：SOCKS5
* 网络内部协议：自定义匿名电路协议

---

## 14.3 MVP 支持范围

第一版建议只支持：

* SOCKS5
* 无认证模式
* `CONNECT`
* 域名 / IPv4 地址
* TCP 流量

后续可扩展：

* 用户名密码认证
* IPv6
* `UDP ASSOCIATE`
* HTTP CONNECT
* TUN / VPN

---

## 14.4 本地 SOCKS5 工作流程

### 步骤 1：接受连接

本地节点监听 `127.0.0.1:1080`。

### 步骤 2：SOCKS5 握手

处理版本协商和认证协商。

### 步骤 3：解析请求

获取：

* 目标主机
* 目标端口
* 命令类型

### 步骤 4：策略路由

本地节点决定：

* 走匿名多跳
* 走低延迟路径
* 走指定 exit
* 是否复用已有 circuit

### 步骤 5：建立 stream

在选中的 circuit 上分配 `stream_id`。

### 步骤 6：发送 BEGIN_TCP

向 exit 发起目标连接。

### 步骤 7：开始双向转发

本地 TCP <-> circuit stream

---

## 14.5 SOCKS5 状态机

```text
SOCKS_ACCEPT
  -> SOCKS_HANDSHAKE
  -> SOCKS_REQUEST_PARSED
  -> ROUTE_SELECTED
  -> CIRCUIT_READY
  -> STREAM_OPENING
  -> STREAM_CONNECTED
  -> RELAYING
  -> CLOSED
```

---

## 14.6 一个 SOCKS 连接与 stream 的关系

建议：

* 一个本地 SOCKS5 连接 = 一个匿名 `stream_id`

多个 stream 可复用同一条 circuit。

示例：

```text
SOCKS conn #1 -> circuit 1001 / stream 1
SOCKS conn #2 -> circuit 1001 / stream 2
SOCKS conn #3 -> circuit 1002 / stream 1
```

---

## 14.7 circuit 池

本地节点应维护 circuit pool：

* `anonymous_pool`
* `low_latency_pool`
* `country_specific_pool`
* `manual_exit_pool`

新请求优先复用已有 circuit，避免每次都重建整条路径。

---

## 15. 路径选择与策略路由

## 15.1 目标

客户端本地根据节点描述、评分和策略规则选择匿名路径。

## 15.2 默认路径

建议默认：

* 匿名模式：`2 relay + 1 exit`
* 低延迟模式：`1 relay + 1 exit`
* 性能模式：`0 relay + 1 exit`

---

## 15.3 路径选择条件

### Relay 选择条件

* 在线时间较长
* 建链成功率高
* RTT 较低
* 带宽较高
* 不与其他 hop 同 ASN / 同地区 / 同运营者

### Exit 选择条件

* 出口国家符合需求
* 允许目标端口
* 成功率高
* 带宽较高
* 信誉正常

---

## 15.4 去相关性规则

为了减少路径关联，路径中的节点应尽量避免：

* 同一 ASN
* 同一运营者
* 同一付款地址
* 同一云厂商
* 同一 /24 子网

## 17. 流控与多路复用

## 17.1 设计目标

* 一条 circuit 复用多个 stream
* 避免一个大流压垮整个 circuit
* 支持窗口更新和 backpressure

## 17.2 方案建议

为每个 stream 维护：

* 发送窗口
* 接收窗口
* 待确认字节数
* 缓冲区大小

为每个 circuit 维护：

* 最大并发 stream 数
* 总体窗口
* 总带宽预算

## 17.3 Backpressure

当某个 stream 下游阻塞时：

* 暂停本地读取
* 触发窗口收缩
* 避免无限缓冲导致 OOM

## 18. 节点评分与信誉系统

## 18.1 目标

完全开放的去中心化节点网络容易被垃圾节点、低质量节点、恶意节点污染，因此客户端需要维护本地评分。

## 18.2 本地评分维度

每个已知节点可维护以下统计：

* 建链成功率
* 平均 RTT
* DATA 丢失率
* 电路中断率
* 平均带宽
* 被拒绝率
* 错误率

## 18.3 评分用途

* 路径选择时加权排序
* 避免频繁使用低质量节点
* 降低恶意节点被选中的概率

## 18.4 共享信誉

v1 不建议强依赖全网共享信誉，只做本地评分。
后续可扩展带签名的观测报告，但只能作弱参考。

---

## 19. 安全设计

## 19.1 目标

尽可能降低节点看到的敏感信息范围，并提升网络鲁棒性。

## 19.2 前向保密

每条电路都必须用临时会话密钥，不依赖长期身份密钥直接加密数据。

## 19.3 DNS 泄漏防护

* 本地 SOCKS5 不得主动解析域名
* 默认使用 exit 侧 remote DNS
* 管理界面中需明确标注

## 19.4 中继可见性控制

Relay 节点只应看到：

* 上一跳 peer
* 下一跳 peer
* 局部 circuit 状态

不能看到：

* 目标地址
* 完整路径
* 其他 hop 的会话密钥

## 19.5 Exit 风险说明

Exit 节点虽然无法直接知道完整路径，但能看到：

* 最终目标地址
* 明文 HTTP 内容
* TLS SNI / 流量元数据

因此系统文档必须提醒：

**真正的端到端安全仍依赖 HTTPS / TLS / 应用层加密。**

## 19.6 流量分析缓解

v1 可选措施：

* 固定大小 cell
* padding
* 随机 keepalive
* 分批发送

但不能承诺完全防止时序分析。

## 19.7 重放攻击防护

通过：

* 单向 seq / counter
* AEAD nonce 管理
* replay window
* key epoch 切换

进行防护。

## 20.1 必要性

## 26. MVP 验收标准

第一版可认为完成的条件：

1. 节点启动后本地成功监听 `127.0.0.1:1080`
2. 节点能加入 DHT 并发现其他 relay / exit
3. 节点能发布自己的 relay descriptor
4. 启用出口的节点能发布 exit descriptor
5. 本地浏览器设置 SOCKS5 后可访问 HTTPS 网站
6. 流量能够经至少 `1 relay + 1 exit` 路径转发
7. Relay 看不到目标域名
8. 域名解析发生在 exit 节点
9. circuit 失败时支持自动重试
10. 节点本地能查看当前 circuit 和 stream 状态

## 27. 风险与注意事项

### 27.1 Exit 节点法律与滥用风险

运营出口节点存在实际风险，需要：

* 端口限制
* 黑名单
* 速率限制
* 日志策略
* 使用条款

### 27.2 Relay 质量不均

默认所有节点都是 relay，会导致大量弱节点混入，需要评分与选路优化。

### 27.3 去中心化发现污染

DHT 中可能被注入垃圾 descriptor，需要签名、TTL 和本地筛选。

### 27.4 多跳带来的性能损耗

多跳匿名不可避免会增加：

* RTT
* CPU 加解密开销
* 内存缓冲压力

需要通过 circuit pool 和少拷贝优化缓解。

## 28. 最终结论

本设计定义了一套完整的：

**去中心化匿名代理覆盖网络协议 + 本地 SOCKS5 Agent 体系**

核心特征为：

* 所有节点默认是 Relay
* 本地始终提供 SOCKS5 入口
* 用户流量通过本地节点接入匿名网络
* Relay 仅知局部路径
* Exit 负责访问公网
* 节点通过 DHT + Gossip 自主发现
* 路径由客户端本地构建与控制
* 协议天然支持多跳、评分、策略路由和后续经济模型扩展

这套架构适合从 Go + libp2p 起步，先做 MVP，再逐步增强匿名性、稳定性和治理能力。
