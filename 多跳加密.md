

# 一、先给 AI 的背景说明

你当前的 `meshproxy` 已经初步实现了：

* 本地 SOCKS5
* relay / relay+exit 模式
* 基础 circuit / stream
* Local -> Exit
* Local -> Relay -> Exit

但是当前版本如果 relay 转发的内容没有做真正的分层加密，那么它只是一个**多级代理链**，不是严格意义上的匿名网络。

现在要做的升级目标是：

## 升级目标

让每个 hop 只知道它自己的局部信息：

* 客户端本地构建电路
* 客户端与每个 hop 独立协商临时会话密钥
* 客户端发送数据时，按 hop 顺序逐层加密
* relay 收到后只解自己这一层
* relay 只能知道：

  * 自己处理的是哪个局部 circuit
  * 下一跳是谁
  * 剩余密文是什么
* relay 不能看到最终目标地址
* exit 才能解开最后一层，得到真正的 `BEGIN_TCP target_host:target_port`

当前项目已经有基础主链路，请不要推翻重写。请只做“多跳分层加密”的增量改造，优先保持现有 relay / exit / socks5 主流程可运行。

## 最终效果

对于路径：

`Local -> Relay -> Exit`

应满足：

* **Relay 看不到最终目标域名/IP/端口**
* **Exit 看不到完整路径**
* **客户端本地掌握整条路径和所有 hop 的会话密钥**
* **返回数据沿相反方向逐层封装**

## 一、总体目标

请把当前的多跳代理链升级为真正的多跳分层加密链路，使其满足：

### 目标 1：每 hop 独立会话密钥

客户端本地与每个 hop 分别建立独立临时会话密钥，而不是让所有 hop 共用一个连接层安全上下文。

### 目标 2：客户端逐层加密

客户端发送数据时，应按路径从最后一跳到第一跳逐层加密。

例如对于：

`Local -> Relay -> Exit`

客户端发送 `BEGIN_TCP(example.com:443)` 时，应做到：

1. 先用 Exit 的 forward key 加密出 inner payload
2. 再用 Relay 的 forward key 包一层 outer payload
3. 发送给 Relay

### 目标 3：Relay 只解一层

Relay 收到后只能：

* 用自己的 hop key 解开最外层
* 得到局部头部
* 知道下一跳是谁
* 将剩余密文转发给 Exit

Relay 不能看到：

* 最终目标 host
* 最终目标 port
* 最终 BEGIN_TCP 明文
* 完整路径

### 目标 4：Exit 解最后一层

Exit 收到后：

* 解开最后一层
* 得到真正的 BEGIN_TCP / DATA / END / WINDOW_UPDATE
* 执行 remote DNS 和公网 TCP 连接

### 目标 5：返回方向也分层

Exit 回包时，应按 backward path 分层封装，使客户端本地逐层解开。

## 四、密钥协商模型

请实现每 hop 独立密钥协商。

### 要求

* 使用 `X25519` 临时密钥交换
* 每个 hop 都要建立独立 session
* 不使用长期身份密钥直接加密业务数据
* 长期身份密钥只用于身份认证和 descriptor 签名
* 使用 HKDF 派生会话密钥

### 每 hop 至少派生两个方向密钥

* `forward key`
* `backward key`

建议派生方式：

* `HKDF(shared_secret, "meshproxy-hop-fwd")`
* `HKDF(shared_secret, "meshproxy-hop-bwd")`

### 每 hop 建议维护

* peer_id
* role（relay / exit）
* ephemeral pubkey
* shared secret
* forward key
* backward key
* seq / counter
* created_at

---

## 五、分层加密模型

请实现 onion-style layered wrapping / unwrapping。

### forward 方向

对路径：

`Local -> Relay -> Exit`

客户端发送业务请求 `BEGIN_TCP(example.com:443)` 时，应执行：

1. 将 `BEGIN_TCP` 编码为 inner relay payload
2. 使用 Exit 的 forward key 加密，得到 `cipher_exit`
3. 再构造 Relay 层 payload，其中包含：

   * 下一跳信息
   * 剩余密文 `cipher_exit`
4. 使用 Relay 的 forward key 加密，得到 `cipher_relay`
5. 把 `cipher_relay` 发送给 Relay

### Relay 收到后的处理

Relay 只能：

1. 使用自己的 forward key 解密最外层
2. 得到：

   * 局部 relay header
   * 下一跳 peer_id
   * inner ciphertext
3. 将 inner ciphertext 发给下一跳 Exit

Relay 不应得到最终 BEGIN_TCP 的明文内容。

### Exit 收到后的处理

Exit：

1. 使用自己的 forward key 解最后一层
2. 得到 BEGIN_TCP / DATA / END / WINDOW_UPDATE 等真正业务指令
3. 执行出口逻辑

---

## 六、返回方向模型

请实现 backward layered wrapping。

### backward 方向

例如 Exit 连接目标成功后返回 `CONNECTED`，或者返回 `DATA` 时：

1. Exit 先用自己的 backward key 封装
2. Relay 收到后不能看到最终客户端上下文，只做局部处理后继续向客户端方向转发
3. 客户端最终逐层解开 backward 数据

对于当前 1 relay + 1 exit 的 MVP，可实现为：

* Exit 用 backward key 封装最后一层
* Relay 再用自己的 backward key 做一层封装
* 客户端逐层解开

如果你认为更合理，也可以让 backward 方向沿既有 circuit 局部状态映射完成，但必须满足：
**客户端能逐层解开，relay 只知道局部。**
